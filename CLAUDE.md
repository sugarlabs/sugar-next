# Sugar Labs Workspace

Multi-repo workspace for modernizing Sugar (the OLPC/Sugar Labs desktop
environment): retargeting it at current Linux laptops, resuming the stalled
GTK4 port, and eventually redesigning its visual language.

Full rules and rationale live in `specbook/docs/base-standards.md` — read it
before doing substantive work here. Short version:

- Small changes, one OpenSpec change at a time.
- Match each repo's existing build system (Autotools vs. modern
  pip/Makefile) — don't introduce a new one as a side effect.
- Code, commits, and specs are in English regardless of what language the
  conversation happens in.
- Environment/setup friction counts as a bug, not a footnote — it's the
  actual reason the last GTK4 porting attempt stalled
  ([details](specbook/docs/gtk-porting-standards.md)).

## Workflow

This workspace uses [OpenSpec](https://github.com/Fission-AI/OpenSpec) as
its spec/change engine:

```
/opsx:explore              # optional — scope out an idea, no artifacts
/opsx:propose <name>       # creates openspec/changes/<name>/
/opsx:apply                # implement tasks.md one item at a time
/opsx:sync                 # merge delta specs into openspec/specs/
/opsx:archive <name>       # validate + archive
```

## Agents

- `sugar-developer` — plans (does not directly implement) work touching
  PyGObject/GTK/D-Bus code across the Sugar repos.

## Skills

- `setup-sugar-workspace` — clone the Sugar repos actually needed for a
  task into `repos/`.
- `writing-skills` — how to add new skills to this workspace.
- `openspec-*` — generated by `openspec init`; drive the propose/apply/
  sync/archive lifecycle.

## Layout

```
CLAUDE.md
openspec/            # OpenSpec engine state (specs, changes, config)
.claude/
  skills/
  agents/
specbook/
  docs/               # standards + reference docs (this is the real content)
repos/                # Sugar repos get cloned here when work starts (empty for now)
```
